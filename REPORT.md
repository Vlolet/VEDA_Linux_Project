# 심화 실습 평가2 (리눅스 프로그래밍) 개발 문서

**제출자:** 이유준  
**프로젝트명:** TCP 기반 원격 하드웨어 제어 시스템  

---

## 1. 프로젝트 개요
본 프로젝트는 Ubuntu 환경의 클라이언트가 TCP 네트워크를 통해 라즈베리파이4 서버에 접속하여, 다양한 하드웨어(LED, Buzzer, CDS, 7-Segment)를 원격으로 제어하고 모니터링하는 시스템을 구축하는 것을 목적으로 합니다. 
서버는 **데몬(Daemon) 프로세스**로 구동되며, 하드웨어 제어 코드는 **공유 라이브러리(.so)** 로 분리되어 프로그램 재시작 없이 기능을 업데이트할 수 있는 핫스왑(Hot-Swap) 아키텍처를 적용했습니다.

## 3. 세부 구현 내용

### 3.1. 서버 아키텍처 (멀티 스레드 & 데몬)
*   **데몬 프로세스화 (`main.c`)**: `fork()` 후 부모를 종료하고, `setsid()`로 새로운 세션을 생성하여 터미널 제어권을 벗어납니다. 파일 디스크립터를 `/dev/null`로 리다이렉션하여 완전한 백그라운드 프로세스로 동작시켰습니다.
*   **네트워크 처리 (`network_server.c`)**: `accept()`로 클라이언트 연결 시, 각 클라이언트마다 독립적인 `client_handler_thread`를 생성하여 블로킹 없이 다중 요청을 처리할 수 있는 기반을 마련했습니다.
*   **syslog 로깅**: 데몬화된 서버의 출력 확인을 위해 모든 시스템 로그(`SYSTEM_LOG`)는 `syslog` 데몬을 통해 `/var/log/syslog`에 기록되도록 구현했습니다.

### 3.2. 하드웨어 제어 및 공유 라이브러리 동적 로드
*   **인터페이스 구조체 (`device_interface.h`)**: 모든 장치 라이브러리는 `DeviceInterface` 구조체(init, control, read, exit)를 필수 심볼(`device_ops`)로 구현하도록 규격화했습니다.
*   **디바이스 매니저 (`device_manager.c`)**: `dlopen()`을 사용해 `libled.so`, `libbuzzer.so` 등을 런타임에 적재합니다. 장치 사용 시점에는 `pthread_mutex_lock`을 통해 스레드 간 동시 접근(Race Condition)을 방지합니다.
*   **장치별 구현 특징**:
    *   **LED**: `softPwm`을 사용하여 클라이언트로부터 0~100 사이의 값을 받아 밝기를 조절.
    *   **CDS (조도 센서)**: `wiringPiI2C` 통신을 사용. 클라이언트 명령에 따라 백그라운드 모니터링 스레드가 구동되며, 설정된 Threshold 이상(어두워짐)이 감지되면 내부적으로 LED On 핸들러를 호출합니다.
    *   **7-Segment**: 배열 비트 매핑을 통해 a~g LED를 제어하며, 카운트다운 스레드 구동 시 1초마다 숫자가 줄어들고 0 도달 시 Buzzer 핸들러를 트리거합니다.

### 3.3. 핫스왑 (Hot-Swapping) 메커니즘
*   **기능 요구사항 충족**: 장치 기능 업그레이드 시 전체 서버의 재시작 없이 파일만 업로드하여 수행 가능하게 구성했습니다.
*   **구현 방법**: 데몬 프로세스가 `SIGUSR1` 시그널을 수신하면 플래그(`g_reload_req`)를 세팅합니다. 메인 네트워크 루프에서 이 플래그를 확인하면 `reload_devices()` 함수를 호출하여 기존 핸들을 `dlclose()`하고 갱신된 `.so` 파일을 다시 `dlopen()`하여 장치 모듈을 라이브 교체합니다.

### 3.4. 클라이언트 (UI 및 시그널 처리)
*   **비동기 UI 구조**: `pthread`를 이용해 서버로부터 오는 센서 데이터(CDS 값, 카운트다운 값)를 수신하는 전용 스레드를 분리하였습니다. 수신된 데이터는 `\033[s` (커서 저장) 와 `\033[u` (커서 복원) 기법을 사용하여 사용자의 입력을 방해하지 않고 화면의 특정 좌표만 업데이트합니다.
*   **시그널 처리**:
    *   **SIGINT (`Ctrl+C`) 무시**: 프로그램 실행 도중 강제 종료되지 않도록 요구사항에 따라 `signal(SIGINT, SIG_IGN)`을 선언하고, 메뉴 입력(q)을 통해서만 소켓과 스레드를 정리하고 안전하게 종료되도록 구현했습니다.
    *   **SIGWINCH (창 크기 조절)**: 터미널 창 크기가 변경될 때 UI가 깨지지 않도록 시그널을 잡아 화면을 Clear하고 다시 그리는 반응형 로직을 추가했습니다.

### 3.5. 빌드 자동화 (`build.sh` & CMake)
*   쉘 스크립트 하나로 환경 체크, 클라이언트 호스트 빌드, 서버용 크로스 컴파일(Raspberry Pi 4 toolchain 적용), 불필요 파일 정리(`temp` 디렉토리 활용), `rsync`를 이용한 라즈베리파이 자동 배포까지 원클릭으로 진행되도록 구성했습니다.
*   이 때 기존의 cmake 명령를 사용하지 못하도록 `CMakeLists.txt`파일을 작성하였고, `build.sh`에서 rsync 이전에 필요한 환경 라이브러리(`wiringPi`)가 있는 지 체크하도록 하여 유저 경험을 고려하였습니다.

## 4. 문제점 및 보완 사항

1. **소켓 통신 동시성 문제 (보완 완료)**
   * *문제점*: CDS 스레드나 카운트다운 스레드가 백그라운드에서 데이터를 클라이언트로 지속 송신(`write`)할 때, 동시에 클라이언트가 다른 명령을 전송하면 패킷 충돌 또는 UI 겹침 현상이 있었습니다.
   * *보완*: 송수신 스레드를 철저히 분리하고, 서버 측에서도 각 디바이스가 `write` 할 때의 조건절을 명확히 하여 패킷 구조체(`ControlPacket`) 규격을 통일해 안정성을 높였습니다.

2. **핫스왑 시 스레드 충돌 위험 (보완 완료)**
   * *문제점*: 특정 장치가 작동 중일 때(예: 카운트다운 스레드 진행 중) `SIGUSR1` 시그널로 `dlclose()`가 호출되면 Segmentation Fault가 발생할 수 있습니다.
   * *보완*: `device_manager.c`에 장치 슬롯별 `pthread_mutex_t`를 도입했습니다. 장치를 사용하는 동안은 `lock_and_get_device()`로 잠금을 걸고, 작업이 끝나면 `release_device()`로 풀어주어 사용 중인 장치의 라이브러리가 중간에 해제되지 않도록 보완했습니다.

3. **향후 개선 가능한 사항**
   * 현재 1:1 Server-Client 구조를 가정하고 `g_client_fd` 전역 변수를 사용하고 있으나, 추후 다중 클라이언트의 동시 접근을 허용하기 위해 FD 리스트 관리 및 Select/Epoll 구조로 리팩토링할 여지가 있습니다.
   * 클라이언트 비정상 종료 시 서버의 `read()`가 -1을 반환하는 예외를 좀 더 세분화하여, 네트워크 끊김 시 자동으로 재연결을 시도하는 Heartbeat 로직을 추가하면 시스템 신뢰성이 더욱 향상될 것입니다.
